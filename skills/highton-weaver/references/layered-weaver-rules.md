# 通用 Weaver 分层封装规则

## 1. 目标

把 Weaver OA 文档中的接口，落成当前项目里的分层代码，而不是只写一个零散的 HTTP 调用。

默认目标链路：

- `Provider`
- `Service`
- `Impl`
- `Test`
- request DTO
- response DTO
- constants/enums

如果目标项目已有固定分层，就优先贴合现有风格，不强行套模板。

## 2. 先识别项目分层

在生成代码前，先看代码库里已经存在什么链路。

优先识别：

- 是否存在 `Provider`、`Facade`、`Gateway`、`Client`
- 是否存在 `Service` 接口
- 实现类是否统一放在 `impl` 包
- DTO 是否拆为 `request/response`
- 常量和枚举是否独立管理
- 测试是否放在 `src/test/java`

如果项目已有明确模式，优先遵循项目模式。

如果项目没有明显模式，使用下面这套默认分层。

## 3. 默认分层职责

### 3.1 Provider

职责：

- 暴露给业务方使用的外层调用能力
- 屏蔽 OA 细节

规则：

- 只保留项目语义的入参和出参
- 不写 HTTP 细节
- 一般只做委托转发

### 3.2 Service

职责：

- 声明业务接口能力
- 作为 Provider 和 Impl 之间的抽象层

规则：

- 签名与 Provider 基本保持一致
- 如果列表和数量天然分离，可以同时声明 `list` 和 `count`

### 3.3 Impl

职责：

- 执行真正的 OA HTTP 调用
- 做项目语义与 OA 语义之间的转换

固定流程通常是：

1. 获取 token
2. 将项目身份字段转换成 OA `userid`
3. 按文档要求决定 `access_token` 放 query 还是 body
4. 组装请求参数
5. 发起 HTTP 请求
6. 解析 OA 返回的 `message`
7. 将 `data` 映射成项目响应 DTO
8. 返回项目统一响应结构

### 3.4 Test

职责：

- 验证封装逻辑是否符合项目规范

规则：

- 生成新 Weaver 接口时，默认要补对应的测试类或测试方法
- 优先复用项目里现有的 Weaver 或第三方接口测试母版
- 优先扩展最近的测试类，而不是新开一套测试风格
- 只有在仓库规则明确禁止、用户明确拒绝、或项目确实没有合理测试承载点时，才允许跳过

## 4. 请求参数映射规则

### 4.1 外层使用项目语义

外层接口不要强迫调用方直接传 OA `userid`，除非当前项目本来就以 `userid` 为标准身份字段。

优先保留：

- 项目已有的用户身份字段
- 分页参数
- 业务筛选参数

然后在实现层转换成 OA 需要的字段。

### 4.2 结构化字段优先强类型

如果 OA 文档里的 `condition`、`filter`、`queryParam` 是结构化对象：

- 优先建强类型 DTO
- 内层嵌套继续拆 DTO
- 字段名优先贴 OA 文档原字段

不要长期把复杂结构停留在裸 `JSONObject` 或 `Map`。

### 4.3 access_token 放置位置必须以文档为准

不要假设所有 Weaver 接口一致。

常见两类：

- 放在 body
- 放在 query

文档怎么定义，就怎么实现。

## 5. 响应封装规则

### 5.1 统一先看 message

通常先检查：

- `message.errcode`
- `message.errmsg`

常见成功判断：

- `errcode == "0"`

### 5.2 data 直接映射 DTO

如果 `data` 是数组：

- 直接反序列化成列表 DTO

如果 `data` 是对象：

- 提取字段后构建响应 DTO

### 5.3 字段名优先贴 OA

如果项目没有自己的命名转换规范，优先保留 OA 原字段名，减少中间映射成本。

## 6. 列表接口特殊规则

列表接口经常分为：

- 列表接口
- 数量接口

如果项目返回对象里需要 `totalCount`，并且 OA 文档也拆成 count/list 两个接口，优先同时接入两者。

默认策略：

1. 调列表接口
2. 如果需要稳定总数，再调数量接口
3. `totalCount` 优先使用 count 结果
4. count 不可用时，再退回当前页条数

## 7. DTO 设计规则

请求 DTO：

- `XxxRequestDTO`

响应 DTO：

- `XxxResponseDTO`

如果项目已有其他命名方式，优先贴合项目现状。

复杂对象优先拆成：

- 主请求 DTO
- 筛选条件 DTO
- 列表项 DTO
- 内嵌返回 DTO

## 8. 常量与枚举规则

如果项目存在统一常量/枚举管理：

- 为新 OA 接口补 URL 常量
- 为新 OA 接口补枚举项

如果项目没有这套结构，就不要为了套模板硬加一层。

## 9. 生成代码时的推荐顺序

1. 从文档抽出请求和响应模型
2. 识别当前项目的分层风格
3. 判断是否复用已有 DTO
4. 新增或修改 request DTO
5. 新增或修改 response DTO
6. 新增或修改 Provider
7. 新增或修改 Service
8. 新增或修改 Impl
9. 按项目需要补 constants/enums
10. 识别最近的测试母版和测试落点
11. 为新增接口方法补对应测试类或测试方法

## 10. 最后检查

- 是否贴合了当前项目的真实分层
- 外层是否只保留项目语义
- 实现层是否承担了 OA HTTP 细节
- `access_token` 放置位置是否符合文档
- `Response.success/fail` 或项目统一响应结构是否一致
- 列表接口是否正确处理了 `totalCount`
- 复杂筛选参数是否已经强类型化
- 是否已经补了与接口能力对应的测试类或测试方法
- 如果没有补测试，是否明确写清跳过原因
