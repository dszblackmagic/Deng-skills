# 交付说明输出规范

## 1. 目标

在完成 Weaver 接口代码生成后，输出一份稳定、简洁、可 review 的交付说明。

这份说明不是重复贴代码，而是帮助阅读者快速确认：

- 生成结果落在哪条链路
- 复用了什么现有结构
- 哪些内容直接来自 OA 文档
- 哪些地方做了项目语义适配

## 2. 输出时机

完成代码修改后立即输出。

如果这次任务只是分析文档、没有真正改代码，也要输出一份精简说明，明确当前只完成到哪一步。

## 3. 最低输出内容

至少覆盖以下项目：

### 3.1 采用的架构链路

明确说明本次采用了哪条链路，例如：

- `Provider -> Service -> Impl`
- `Controller -> Service -> Client`
- `Service -> Gateway`

### 3.2 参考了哪个相似接口

明确说明：

- 复用了哪个现有接口或实现作为母版
- 如果没有找到合适母版，也要说明是全新落地

### 3.3 变更文件

列出新增或修改的关键文件，不必展开到每一行。

优先列：

- 外层接口
- service
- impl
- DTO
- constants/enums
- test

### 3.4 文档直映射内容

明确说明哪些内容是直接按 OA 文档落地的：

- API path
- HTTP method
- 请求参数
- 返回字段
- 分页结构
- count/list 关系

### 3.5 项目语义转换

明确说明哪些地方做了转换，而不是照抄 OA 文档：

- 项目用户标识字段转换为 OA `userid`
- 项目统一响应包裹
- DTO 命名适配
- 常量/枚举收敛

### 3.6 token 放置方式

明确写清：

- `access_token` 在 query
- 或 `access_token` 在 body

这点必须显式说明，避免 reviewer 误判。

### 3.7 测试处理

明确写清：

- 已补哪些测试类和测试方法
- 未补测试，因为仓库规则限制
- 未补测试，因为用户明确拒绝
- 未补测试，因为当前项目没有合理测试承载点

## 4. 推荐输出格式

建议使用下面这种紧凑结构：

1. 架构链路
2. 仿写来源
3. 变更文件
4. 文档映射
5. 项目适配
6. 测试情况

## 5. 写作要求

- 用短句，不写长篇解释
- 先说结果，再说原因
- 只保留 review 真正关心的信息
- 不把代码大段重复到说明里
- 不要省略关键决策点

## 6. 示例骨架

可按这个骨架输出：

1. 架构链路：本次采用 `Provider -> Service -> Impl`，因为项目里已有同类接口按这条链路组织。
2. 仿写来源：优先参考了现有的 `XxxProvider / XxxService / XxxServiceImpl`。
3. 变更文件：新增或修改了 provider、service、impl、request DTO、response DTO，以及相关 constants/enums。
4. 文档映射：接口路径、请求方式、分页参数、返回字段和 count/list 关系按 OA 文档直接落地。
5. 项目适配：外层继续使用项目用户标识，在实现层转换为 OA `userid`；响应继续使用项目统一包装。
6. 测试情况：已补 `XxxTest` 中的 `testXxx`；如果未补，明确写原因，不要只写“未要求”。
