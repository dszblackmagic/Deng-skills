# 相似接口优先仿写策略

## 1. 目标

在目标项目里接入新的 Weaver OA 接口时，优先找到最接近的现有接口作为母版。

不要一上来就新建一整套全新结构。

## 2. 为什么优先仿写

好处：

- 更贴近项目现有风格
- 更容易复用 DTO 和通用逻辑
- 减少命名漂移
- 降低 reviewer 的理解成本
- 更容易避免漏改链路文件

## 3. 先找哪类相似对象

优先级从高到低：

1. 同样是 Weaver 接口的现有实现
2. 同样是第三方 HTTP 封装的现有实现
3. 同一模块下功能最接近的接口链路
4. 只有在前面都没有时，才自己新开形状

## 4. 相似度判断维度

优先找下面这些维度相似的现有接口：

- 同样是列表接口
- 同样有分页参数
- 同样有 `count + list` 双接口
- 同样有复杂 `condition/filter`
- 同样需要把项目身份字段映射到第三方身份字段
- 同样返回数组或嵌套对象
- 同样走 GET 或 POST
- 同样把 token 放 query 或 body

## 5. 找到相似接口后怎么处理

### 5.1 先复用链路

优先复用已有的：

- 外层接口组织方式
- Service 和 Impl 的职责分配
- DTO 包路径
- 常量和枚举组织方式
- 日志风格
- 错误处理方式

### 5.2 只替换文档驱动的部分

只改这些内容：

- URL
- 请求字段
- 响应字段
- 成功判断后的数据解析
- 特有的分页或 count 逻辑

不要顺手重构不相关风格。

### 5.3 判断能否直接扩展现有 DTO

先判断：

- 是否已有同名或近似的 request DTO
- 是否已有通用的 filter DTO
- 是否已有通用的列表响应 DTO
- 是否已有可复用的嵌套子 DTO

如果只有少数字段差异：

- 优先扩展已有 DTO
- 或沿用已有字段模式继续新增

不要为了几个字段差异重造整个模型体系。

## 6. 什么时候必须新建

只有这些情况再新建新的接口链路或 DTO：

- 现有接口的职责明显不同
- 现有 DTO 复用会造成语义污染
- 新接口的请求或响应结构差异过大
- 现有链路会让命名变得误导

## 7. 推荐动作顺序

1. 搜索现有 Weaver 或第三方相似接口
2. 选一个最接近的母版
3. 列出要复用的文件和模式
4. 列出只需要替换的文档驱动部分
5. 再决定哪些类真正需要新建

## 8. 最后检查

- 这次生成的结构是不是明显像项目里已有代码，而不是像另一套外来的模板
- 有哪些地方复用了现有 DTO、链路或错误处理
- 新建的类是不是确实无法复用现有类后才新增的
