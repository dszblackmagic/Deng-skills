# PRD Test Execution

## 强制首响门禁

调用本 skill 后，第一响应必须先做执行前确认，不得直接打开页面、调用浏览器工具、字段采集、失败取证或录制视频。

用户单独调用本 skill 时，默认 `01-模块拆解.md`、`02-测试用例.md`、`03-执行清单.md`、`04-测试数据与账号.md` 已经由用户确认，不再询问“是否确认”。只需要检查这些执行必备资料是否存在、可读取、可用于执行。

如果用户本轮没有明确给出完整执行前置条件，必须先逐步询问并等待答复。每次只询问当前顺序中的一个前置条件；用户答复后，再判断下一项是否仍缺失并继续询问。不得一次性把所有缺失项列给用户要求补充。

按以下顺序确认：

1. 执行必备资料是否存在：`01-模块拆解.md`、`02-测试用例.md`、`03-执行清单.md`、`04-测试数据与账号.md`。能在当前上下文或工作区定位时直接检查，不询问确认状态；无法定位时只询问资料目录或缺失文件路径。
2. 测试环境地址和测试路径，测试路径可以是菜单路径或 URL。
3. 登录方式：使用指定测试账号、复用当前登录态，或无需登录；指定账号时还要确认账号、密码获取方式和权限范围。
4. 执行模式：单 agent 独立执行，或多 agent 并行执行。未确认执行模式前禁止执行。
5. 高风险动作授权边界：允许真实提交，或仅验证入口、展示、校验、二次确认和提示。

必备资料存在性检查优先使用当前上下文、用户提供的目录和本地文件检索完成；只有无法定位或文件缺失时，才向用户询问资料目录或缺失文件路径。

优先使用单步句式询问当前缺失项：

```text
执行前先检查必备资料。我当前没有定位到 `01-模块拆解.md`、`02-测试用例.md`、`03-执行清单.md`、`04-测试数据与账号.md`，请提供这些文件所在目录或缺失文件路径。
```

如果用户一次性主动提供了多项信息，可以直接记录这些已确认项，但下一次提问仍只问当前尚缺的下一项。

只有当执行必备资料存在且用户已经明确提供所有执行前置条件时，才汇总最终执行方案。最终执行方案至少包含：必备资料定位结果、测试环境和路径、登录方式和账号处理、执行模式、高风险动作授权边界、优先级执行范围、证据目录。汇总后才能进入依赖检查和页面执行。

## 目标

基于默认已确认且实际存在的 `01-04` 规划资料执行模块自动化验证，按用例优先级目录记录结果，并对失败进行分流取证。此 skill 不负责重新设计测试用例；发现规划缺口时先阻塞并要求回到规划阶段补齐。

执行不是静默后台任务。进入页面前必须按顺序向用户确认测试路径、登录方式、执行模式和风险授权，并在全部确认后汇总最终执行方案；执行过程中必须按阶段输出当前动作、发现、阻塞和证据落盘位置。

## 执行前门禁

开始页面验证前必须确认：

1. `01-模块拆解.md`、`02-测试用例.md`、`03-执行清单.md`、`04-测试数据与账号.md` 已存在且可读取；默认这些资料已由用户确认，不再询问确认状态。
2. 测试环境地址和测试路径已确认，测试路径可以是菜单路径或 URL。
3. 登录方式已确认：指定测试账号登录、复用当前已登录状态，或无需登录直接访问。
4. 执行模式已确认：单 agent 独立执行，或多 agent 并行执行。
5. 高风险动作授权边界已确认。

### 主动询问要求

门禁不是内部推断，但 `01-04` 资料确认状态是默认事实，不向用户询问。只要以下执行前置条件存在缺口，就必须按顺序逐步询问并等待答复，不得进入页面执行、字段采集、失败取证或录制视频：

1. `01-04` 执行必备资料是否存在、可读取；无法定位时只询问资料目录或缺失文件路径。
2. 测试环境地址和测试路径。
3. 登录方式；选择指定账号登录时，还必须确认账号、密码获取方式和权限范围。
4. 执行模式：单 agent 独立执行，或多 agent 并行执行。
5. 高风险动作授权边界：允许真实提交，或只验证入口、展示、校验、二次确认和提示。

### 逐步询问流程

逐步询问必须遵守：

1. 每次只问一个前置条件；不要把多个缺失项合并到同一条消息里。
2. 按固定顺序从上到下推进；前一项没有明确结果时，不询问后一项。
3. 用户主动补充了后续项时，可以记录为已确认，但仍回到顺序中最靠前的缺失项继续问。
4. 不询问 `01-04` 是否已确认；只检查存在性。无法定位资料时，询问资料目录或缺失文件路径。
5. 登录方式选择“指定测试账号”时，把账号、密码获取方式、权限范围作为登录方式的子确认，未完整前不得进入执行模式询问。
6. 全部确认后，先输出最终执行方案汇总，再进入依赖检查；如果仍有不确定项，停留在待确认状态。

单步询问示例：

```text
执行前先检查必备资料。我当前没有定位到 `03-执行清单.md`，请提供该文件路径或 `01-04` 资料所在目录。
```

```text
必备资料已定位。继续确认测试范围：请提供测试环境地址，以及进入被测模块的菜单路径或 URL。
```

```text
登录方式是指定测试账号。请继续确认账号、密码获取方式和权限范围。
```

全部确认后的最终执行方案示例：

```text
前置条件已确认，最终执行方案如下：
1. 必备资料：01-04 已定位且可读取，默认已由用户确认。
2. 测试范围：环境为 xxx，路径为 xxx。
3. 登录方式：复用当前登录态。
4. 执行模式：单 agent 独立执行，按 P0 -> P1 -> P2 -> P3 推进。
5. 风险边界：不执行真实提交，仅验证入口、展示、校验、二次确认和提示。
6. 证据目录：执行结果/P0-P3/。
接下来先做依赖检查，再开始页面执行。
```

未满足门禁时，不进入页面执行。用户尚未答复时停留在待确认状态；只有当用户明确表示无法提供、拒绝确认或要求停止时，才把阻塞原因写入对应优先级目录的 `05-执行记录.md`。无法归属优先级时写入根目录汇总 `05-执行记录.md`；如果目录尚未创建，先创建最小记录文件再写入阻塞原因。

## 过程输出规范

执行过程中必须适当输出，不允许长时间静默。输出要短、具体，面向用户说明正在做什么和已经发现什么。

必须输出的节点：

1. 门禁检查后：说明已确认项、当前正在确认的单个缺失项和下一步；全部确认后输出最终执行方案。
2. 依赖检查后：说明 `chrome-devtools-mcp`、`playwright-cli` 是否可用；不可用时说明降级策略。
3. 每个优先级开始前：说明即将执行的优先级、用例范围和证据目录。
4. 每完成一组关键用例或约 30 秒无用户可见输出时：给出简短进度。
5. 发现字段差异、失败、阻塞或高风险未授权时：立即说明分类、影响用例和将写入的记录文件。
6. 每个优先级收口时：说明通过、失败、阻塞、待确认数量，以及截图、视频、脚本目录。
7. 最终汇总后：说明根目录 `05-执行记录.md`、`06-缺陷记录.md` 是否已更新。

输出示例：

```text
P1 开始执行：覆盖关键查询、列表字段和详情入口，结果会写入 执行结果/P1/05-执行记录.md。
```

```text
发现字段差异：列表缺少“审批状态”，先记录为待确认字段差异，并保存截图到 执行结果/P1/screenshots/。
```

避免输出浏览器内部噪声、完整 Console/Network 原文或大段日志；只摘要关键结果，并把完整证据落盘后在记录中引用。

## 依赖检查

执行前检查当前环境是否具备 `chrome-devtools-mcp` 和 `playwright-cli`。

### chrome-devtools-mcp

按顺序判断：

1. 当前会话可用工具中存在 `mcp__chrome_devtools__` 时，视为可用。
2. 否则运行 `codex mcp list`，检查是否存在 `chrome-devtools` 配置。
3. 仍不可用时，先读取官方仓库或官方文档确认最新安装方式，再询问用户是否安装；不得自动安装。

当前已知的 Codex 推荐命令：

```bash
codex mcp add chrome-devtools -- npx chrome-devtools-mcp@latest
```

需要全局 CLI 时可使用：

```bash
npm i chrome-devtools-mcp@latest -g
```

官方地址：https://github.com/ChromeDevTools/chrome-devtools-mcp

### playwright-cli

按顺序判断：

1. 运行 `command -v playwright-cli`。
2. 不存在时运行 `npm list -g @playwright/cli --depth=0`。
3. 仍不可用时，先读取官方仓库或官方文档确认最新安装方式，再询问用户是否安装；不得自动安装。

当前已知安装命令：

```bash
npm install -g @playwright/cli@latest
playwright-cli install --skills
```

官方地址：https://github.com/microsoft/playwright-cli

## 执行目录

按用例优先级创建并写入：

- `执行结果/P0/`
- `执行结果/P1/`
- `执行结果/P2/`
- `执行结果/P3/`

每个优先级目录必须包含：

- `05-执行记录.md`
- `06-缺陷记录.md`
- `screenshots/`
- `videos/`
- `scripts/`

根目录 `05-执行记录.md` 和 `06-缺陷记录.md` 是最终汇总产物，只在各优先级执行完成或阶段性收口后生成或更新。

## 执行规则

1. 页面执行前必须已经完成强制首响门禁、必备资料存在性检查、逐步主动询问和最终执行方案汇总；用户未答复时停在待确认状态。
2. 单 agent 独立执行时，默认按 `P0 -> P1 -> P2 -> P3` 执行，但仍写入各自优先级目录。
3. 多 agent 并行执行时，每个 agent 只能写自己负责的优先级目录，不直接写根目录汇总文件。
4. 已登录时复用当前会话，不重复登录；未登录且登录方式为指定账号时再执行登录。
5. `P0` 验证页面进入、核心入口和主链路；`P1` 验证关键场景和关键字段；`P2` 验证异常、权限和边界；`P3` 仅做低频或观察性补充。
6. 通过用例不截图，只在执行记录中写实际结果摘要。
7. 缺账号、缺权限、缺数据、环境不稳定或依赖受限时，标记为“阻塞”，默认不登记缺陷。
8. 高风险动作未授权时，只验证入口、展示、校验、二次确认或提示，不执行真实提交。
9. 每个优先级目录的执行记录要写入本轮确认信息，包括测试路径、登录方式、执行模式和高风险授权状态。
10. 最终执行方案是进入依赖检查和页面执行前的固定交付物；不得用内部推断或零散确认替代。

## 字段一致性比对

默认比对查询区、列表表头、详情区、关键操作按钮和关键弹窗字段。字段差异包括缺少、多出、命名不一致和语义疑似不一致。

字段差异先记录到所属优先级目录 `05-执行记录.md`。确认属于产品设计或实现问题后，再写入同目录 `06-缺陷记录.md`，最终汇总到根目录 `06-缺陷记录.md`。

字段类失败只截图，不录视频。

## 失败分流

字段类失败：

1. 截图。
2. 记录预期字段、实际字段和差异分类。
3. 关联 `F-xxx` 功能编号、字段区域或用例编号。

操作类失败：

1. 首轮截图。
2. 采集 Console 摘要。
3. 采集 Network 摘要。
4. 使用 `playwright-cli` 重跑并录制视频。

如果 `playwright-cli` 不可用，不阻塞整体执行；在执行记录中说明依赖不可用，并退化为截图、Console 和 Network 证据。

## 完成标准

至少满足：

1. 已完成强制首响门禁，且没有询问 `01-04` 是否已确认；只检查执行必备资料是否存在、可读取。
2. 已明确询问并获得执行模式选择：单 agent 独立执行，或多 agent 并行执行。
3. 已在必备资料定位完成、执行参数确认后汇总最终执行方案，再进入依赖检查和页面执行。
4. 执行过程中已经按过程输出规范向用户同步关键阶段、阻塞、失败和证据位置。
5. 已执行或阻塞记录 `P0/P1` 用例。
6. 已执行的优先级目录均存在独立 `05-执行记录.md` 和 `06-缺陷记录.md`。
7. 字段差异、失败、阻塞或高风险提示证据已落盘并在记录中引用。
8. 根目录汇总 `05-执行记录.md` 已更新。
9. 若存在确认缺陷，根目录汇总 `06-缺陷记录.md` 已更新。
